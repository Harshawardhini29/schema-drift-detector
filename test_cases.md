# Schema Drift Detector - Test Case Document

This document lists the test cases for the **Schema Drift Detector** application, covering both frontend interactions and backend services.

## Test Case Table

| Test ID | Module | Test Case Description | Preconditions | Steps | Expected Result | Severity |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **TC-AUTH-01** | Authentication | Successful User Login | User is registered in the database. | 1. Navigate to `/login`<br>2. Enter valid email and password<br>3. Click "Login" | User is successfully authenticated, JWT token is saved, and user is redirected to the `/` Dashboard. | High |
| **TC-AUTH-02** | Authentication | Failed Login with Invalid Credentials | User enters wrong password. | 1. Navigate to `/login`<br>2. Enter registered email and invalid password<br>3. Click "Login" | Error message "Invalid credentials" is displayed. User remains on the login page. | High |
| **TC-AUTH-03** | Authentication | Route Protection | User is not logged in. | 1. Attempt to access `/connect`, `/scan`, or `/reports` directly via URL | User is blocked and automatically redirected to `/login` page. | High |
| **TC-CONN-01** | Connection Management | Add Postgres Connection | Backend and PostgreSQL DB are running. | 1. Navigate to `/connect`<br>2. Fill in database credentials (alias, connection string)<br>3. Click "Save Connection" | Database connection is validated, saved to `localStorage`, and appears in the connections list. | High |
| **TC-CONN-02** | Connection Management | Test Connection (Success) | Valid connection string. | 1. Navigate to `/connect` or `/scan`<br>2. Input valid database connection string<br>3. Click "Test Connection" | Success alert/badge "Connection Successful" is shown. | Medium |
| **TC-CONN-03** | Connection Management | Test Connection (Failure) | Invalid connection string or offline DB. | 1. Input invalid host/credentials in connection string<br>2. Click "Test Connection" | Error message indicating the connection timeout or credentials issue is displayed. | Medium |
| **TC-SCAN-01** | Scan Engine | Run Scan & AI Analysis | At least one database connection is saved. | 1. Navigate to `/scan`<br>2. Select a preconfigured connection<br>3. Click "Run AI Scan" | Backend extracts schema, compares it with the previous snapshot, runs AI analysis, and renders the result cards with risk levels and AI summaries. | Critical |
| **TC-SCAN-02** | Scan Engine | Manual SQLite Upload Scan | Valid SQLite `.db` file. | 1. Navigate to `/scan`<br>2. Select SQLite DB Type<br>3. Upload a local SQLite `.db` file<br>4. Click "Run AI Scan" | SQLite file is uploaded, parsed, scanned for drift, and comparison results are displayed successfully. | High |
| **TC-DIFF-01** | Diff Engine | Column Added Detection (Additive) | A column was added to the schema since the last snapshot. | 1. Add a new column to a target database table<br>2. Trigger a schema scan | The Diff Engine detects the added column, classifies it as **Additive** with **Low Risk**, and shows a `+` symbol in the UI. | Critical |
| **TC-DIFF-02** | Diff Engine | Column Removed Detection (Breaking) | A column was deleted from the schema. | 1. Drop a column from a target database table<br>2. Trigger a schema scan | The Diff Engine detects the removed column, classifies it as **Breaking** with **High Risk**, and highlights it in red. | Critical |
| **TC-DIFF-03** | Diff Engine | Data Type Altered Detection | A column's data type length was changed (e.g. VARCHAR(100) -> VARCHAR(120)). | 1. Alter a column data type length in the DB<br>2. Trigger a schema scan | The Diff Engine detects the type mismatch, classifies it as **Potentially Breaking** with **Medium Risk**. | Critical |
| **TC-DIFF-04** | Diff Engine | Nullable Mismatch Detection | A column nullable constraint was changed to NOT NULL. | 1. Alter a column constraint to NOT NULL<br>2. Trigger a schema scan | The Diff Engine detects the constraint change and classifies it as **Breaking** (High Risk) because it can break existing API inserts. | Critical |
| **TC-AI-01** | AI Analysis | Generate LLM Summary & Suggestions | Schema drift is detected. | 1. Run a scan containing drifts | AI returns a coherent executive summary, identifies tables impacted, and provides specific remediation/mitigation advice. | High |
| **TC-SNAP-01** | Snapshot History | View Snapshots List | Snapshots exist in the database. | 1. Navigate to `/snapshots` | The page displays a list of all historical schema captures sorted by timestamp, showing database alias and table counts. | Medium |
| **TC-SNAP-02** | Snapshot History | Inspect Snapshot Schema | Snapshots exist. | 1. Navigate to `/snapshots`<br>2. Click on a snapshot card | The card expands to show a nested accordion list of all tables, columns, data types, and primary keys captured in that snapshot. | High |
| **TC-SNAP-03** | Snapshot History | Toggle Auto-Capture | Database connections list is populated. | 1. Navigate to `/snapshots`<br>2. Check "Enable Auto-Capture" on a database<br>3. Select frequency (e.g., 30 mins) | Settings are stored. A background worker periodically triggers silent schema extractions while the app is active. | Medium |
| **TC-REP-01** | Reports Visualizer | List Drift Reports | At least one drift report exists. | 1. Navigate to `/reports` | A list of all historical schema drift events is rendered, showing timestamp, change count, database name, and overall risk badge. | High |
| **TC-REP-02** | Reports Visualizer | View Report Details | Reports exist. | 1. Navigate to `/reports`<br>2. Click on a drift report card | The full report is opened, showing the AI Executive Summary, stats breakdown (number of breaking/additive changes), and grouped tables. | High |
| **TC-REP-03** | Reports Visualizer | Navigate Back | On a report detail page. | 1. Click "Back to Reports" button | The application correctly routes the user back to the `/reports` overview page. | Low |
| **TC-NAV-01** | Navigation | Responsive Sidebar | Screen width varies. | 1. Resize viewport from Desktop to Mobile | The sidebar adjusts responsiveness (minimizes/toggles overlay) ensuring pages remain fully viewable. | Medium |
